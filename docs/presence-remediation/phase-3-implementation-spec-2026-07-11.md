## Decisions

- **D1 — Three canonical migrations, one phase commit**: (A) idempotency + supporting schema, (B) runtime-control write gate, (C) `transition_user_location` + snapshot + space-deletion hardening. Same transient-ownership pattern as Phase 2 (grant→alter→revoke, catalog asserts final zero).
- **D2 (revised) — knock_requests schema hardening was already completed in Phase 1** (`20260710121000_knock_requests_hardening.sql`: all 7 columns, backfill+expire, NOT NULL, `consumed` status, state-machine CHECK, partial-unique/rate indexes, FK RESTRICT). Phase 3 makes NO knock schema change; only Phase 4's create/respond/expire functions remain. `transition_user_location` consumes grants against the existing hardened columns; DB tests insert synthetic approved rows to exercise `knock-enter` (feature stays 503-disabled).
- **D3 — Mode check inside the transition fn**: when `private.presence_runtime_control.mode <> 'atomic'`, `transition_user_location` deletes its still-null idempotency claim and returns uncached retryable `PRESENCE_MAINTENANCE` with no side effect. The write-gate trigger stays the authoritative backstop (L812/L815: atomic markers valid only in mode `atomic`). Staging stays `legacy`; the new route ships non-operational until the Phase 10 cutover.
- **D4 — Repair + unique index are scripts, not migrations**: `scripts/presence-open-log-repair.sql` (CTE from L774-801, SELECT-first review shape) and the `ux_space_presence_log_one_open_per_user` index DDL (L836-839). Executed only by the local cutover-rehearsal test until Phase 10 authors the final CLI migration.
- **D5 — Full cutover rehearsal in DB tests**: legacy → `enter_presence_maintenance` → repair script → unique index → `activate_atomic_presence_writer` → atomic transitions; plus `enter_atomic_presence_maintenance` incident rehearsal (drain/rollback of in-flight atomic writers, roll-forward only via activate, never to legacy). Exit-gate races (L1565-1577) run against the activated-atomic local DB.
- **D6 — Legacy writer ledger wiring now**: every inventoried legacy location/offline handler (per `phase-0-writer-caller-inventory-2026-07-10.md`) calls `begin_legacy_presence_write` before its first presence read/write and `end_legacy_presence_write` in `finally`. Maintenance ⇒ 503 `PRESENCE_MAINTENANCE`; atomic ⇒ 426 `CLIENT_UPGRADE_REQUIRED` without beginning. Old route body/behavior otherwise untouched (no translation into the new RPC).
- **D7 — Drop `remove_user_from_all_spaces`**: Phase 0 catalog proves it is a live NO-OP (empty body, "logic deprecated"). Migration C drops it; the sole caller (`SupabaseUserRepository.updateLocation`) loses the invocation in the same diff.
- **D8 — purge v2**: replace Phase 2 `purge_presence_history` with v2 that additionally deletes completed non-logout `location_transition_requests` rows older than 30 days, retains a completed logout row while its auth-session fence exists, and never deletes a null-result row (L757). Reschedule the daily cron to v2 in the same migration.
- **D9 — Space deletion hardening**: placement/session/log/knock FKs referencing `spaces` become `ON DELETE RESTRICT`; revoke direct `authenticated` DELETE on `public.spaces`; the server deletion route maps FK restriction to 409 `SPACE_IN_USE` when any placement/session/log/retained-knock reference exists. Closed audit history is a reference on purpose — no cascade.
- **D10 — Deferred**: `remove_company_member_and_presence` (function + route) = Phase 7. Client coordinator/426 reload handler = Phase 5. Users-write audit = Phase 6. AuthContext logout wiring = Phase 7.
- **D11 — Reason enum lives in one place** (`location_transition_requests.reason` CHECK + fn validation, L735-746); route Zod schema mirrors it. `expectedLocationVersion` required iff `auto-*`/`knock-enter` (null otherwise) enforced at both route and fn.

- **D12 — Implementation deviations accepted in review**: (a) rejoin evidence window concretized as `coalesce(retired_at, expires_at) >= operation_time - 5 minutes` (handoff says only "recent"; revisit in Phase 8 if UX needs longer grace); (b) log-close timestamps use `greatest(op_time, entered_at + 1 microsecond)` for strict ordering; (c) `get_company_presence_snapshot` is a plpgsql wrapper around ONE materialized-CTE statement so it can raise typed `PRESENCE_SNAPSHOT_TOO_LARGE` / `PRESENCE_VIEWER_NO_COMPANY`; (d) transition new-log rows use `session_type 'workspace'`, `context 'atomic-transition'`.

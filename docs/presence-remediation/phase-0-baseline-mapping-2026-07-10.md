## Baseline source and method

- Source: `supabase db dump --schema public,private` against live project `vhabpcoyypobgasacsko` (schema-only, **zero data rows**, no secrets/auth identities/storage objects).
- The dump reproduces the *current* live state, which already includes the 8 canonical `supabase/migrations`. To order the baseline **before** those migrations without collisions, statements that a canonical migration recreates **non-idempotently** were removed from the baseline (they get created when the migration replays on top).
- Statement-level filtering (dollar-quote-aware split), then `supabase db reset` iterated until clean.

## Added to baseline: realtime publication membership

Live `supabase_realtime` has 9 tables; `supabase db dump --schema` omits publication membership (database-level object). The canonical messaging migration adds 4 (messages, conversations, message_reactions, message_read_receipts). The other 5 — **spaces, users, knock_requests**, message_attachments, conversation_members — were added to live outside any tracked migration. An idempotent `DO` block at the end of the baseline re-adds those 5 so local presence realtime matches prod.

## Exit-gate evidence

- `supabase db reset` succeeds **twice consecutively** from committed files with pinned CLI 2.109.1 (Docker). The `error running container: exit 1` seen once was the Realtime container in a stale state during `Initialising schema`; a full `supabase stop --no-backup && supabase start` cleared it and both subsequent resets printed `Finished supabase db reset`.
- `npm run test:presence-db` reaches Postgres and passes (2/2 harness assertions) — proves the presence DB command reaches the DB.

## Boundary verifications (local, disposable)

- **pg_cron**: available 1.6, `create extension pg_cron` succeeds locally; `cron` schema owned by `supabase_admin` (local superuser). On hosted Supabase cron jobs run as `postgres` — to be reconfirmed on staging before the Phase 2 maintenance migration.
- **presence_maintenance_owner**: creatable and reads back as `NOLOGIN NOINHERIT NOBYPASSRLS` (rolcanlogin=f, rolinherit=f, rolbypassrls=f). Role + extension dropped after the check.
- **auth.sessions**: local shape matches live (id, user_id, not_after, refreshed_at, …). Server-side exact-session absence check (`select 1 from auth.sessions where id = <session_id>`) is feasible; `postgres` holds SELECT. `supabase-js` 2.97 supports `signOut({ scope: 'local' })`. No stop condition on auth-session verifiability.

## Residuals (not blocking Phase 0)

- Exact JWT lifetime / session timebox / inactivity / single-session Auth settings: live in the Supabase dashboard Auth config (not SQL-queryable). Needed to tune Phase 2 lease expiry; read from dashboard before Phase 2. The `session_id` claim itself is a standard GoTrue access-token claim.
- pg_cron "runs as postgres" reconfirmation on staging (local uses supabase_admin).
- Migration-history reconciliation: live `schema_migrations` has only 4 entries and none map to the 8 canonical files. A `supabase migration repair`/history plan is required before any `supabase db push` to prod; the local baseline does not touch prod history.

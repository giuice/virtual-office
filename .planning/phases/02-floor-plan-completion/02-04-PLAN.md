<objective>
Close the remaining Phase 2 access-control gap by moving restricted-space authorization into the server boundary for `/api/users/location`, while preserving the 5-minute reconnection behavior for users rejoining a private space they were already occupying.

Purpose: The current client-side knock UX is implemented, but the generic location API still lets a caller set `current_space_id` directly. This plan makes the server the source of truth for private-room entry, ties the route to the authenticated Supabase user, reuses the existing `knock_requests` approval state instead of inventing a second access system, and adds regression coverage for both FLOR-01 and the FLOR-04 interaction.

Output: Hardened `users/location` route plus targeted automated tests proving that unauthorized private joins fail, approved/private and grace-period joins succeed, and approved-knock authorization is recorded and consumed.
</objective>

Update `src/app/api/users/location/route.ts` so the route stops trusting the client for identity or private-space authorization.

Concrete implementation requirements:

1. **Authenticate the request with Supabase server auth before reading the target user row**
   - Create the normal server client with `createSupabaseServerClient()`
   - Call `const { data: authData, error: authError } = await supabase.auth.getUser()`
   - If `authError` or `!authData.user`, return `401` JSON with `code: 'UNAUTHORIZED'`
   - Create a service-role client with `createSupabaseServerClient('service_role')` for the cross-table authorization checks
   - Resolve the caller's app user through `new SupabaseUserRepository(supabaseAdmin).findBySupabaseUid(authData.user.id)`
   - Never compare `authData.user.id` to `users.id`; only compare it to `users.supabase_uid`

2. **Reject mismatched `userId` payloads**
   - Keep the existing payload shape for compatibility with `useUserPresence` and `useLastSpace`
   - After resolving the authenticated app user, compare `validationResult.data.userId` to `authenticatedUser.id`
   - If they differ, return `403` JSON with `code: 'USER_MISMATCH'` and do not update the location

3. **Fetch the target space with all fields needed for server-side validation**
   - When `spaceId` is non-null, query `spaces` for `id, company_id, status, capacity, access_control`
   - Reject cross-company moves with `403` and `code: 'CROSS_COMPANY_SPACE'`
   - Preserve the existing full-space `409` behavior with `code: 'SPACE_FULL'`
   - Preserve the existing status guard so non-`active` and non-`available` spaces are rejected before location mutation

4. **Enforce restricted-space authorization on the server**
   - Treat the space as restricted when `access_control?.isPublic === false`
   - Allow direct entry without extra checks only when at least one of these is true:
     - `authenticatedUser.role === 'admin'`
     - `authenticatedUser.currentSpaceId === spaceId`
     - there is a matching approved knock row in `knock_requests`
     - there is a recent prior occupancy row in `space_presence_log` showing the same user exited that same space within the last 5 minutes
   - For the approved-knock lookup, query `knock_requests` with these exact filters:
     - `.eq('space_id', spaceId)`
     - `.eq('requester_id', authenticatedUser.id)`
     - `.eq('status', 'approved')`
     - `.eq('decision', 'APPROVE')`
     - order by `updated_at` descending and take the newest row
   - For the grace rejoin lookup, query `space_presence_log` for the newest row matching `user_id = authenticatedUser.id` and `space_id = spaceId` where `exited_at` is not null, then allow if `Date.now() - new Date(exited_at).getTime() < 5 * 60 * 1000`
   - If none of the allowed paths apply, return `403` JSON with `code: 'SPACE_ACCESS_DENIED'` and an error message that explicitly mentions a private space requires approval or recent occupancy

5. **Record and consume approved-knock authorization**
   - Extend `syncSpacePresenceLog` to accept an `authorizedByUserId?: string | null`
   - When inserting a new `space_presence_log` row for a restricted-space join authorized by knock approval, include `authorized_by: responder_id`
   - After a successful join that used an approved knock row, delete that specific `knock_requests.id` row with the service-role client so old approvals cannot be replayed forever
   - Do not delete rows for admin entry, same-space entry, or grace-period rejoin

6. **Use the authenticated user record as the source of truth for the mutation**
   - Derive `previousSpaceId` from `authenticatedUser.currentSpaceId`
   - Call `userRepository.updateLocation(authenticatedUser.id, spaceId)` rather than trusting a payload user ID for the mutation target
   - Keep the existing offline cleanup flow, but ensure the offline update also applies to `authenticatedUser.id`

Three targeted changes to useUserPresence.ts:

**1. Add isPresenceReady flag**
Add a `const [isPresenceReady, setIsPresenceReady] = useState(false);` state variable. Set it to `true` inside the first `presence` `sync` event handler (the `handlePresenceSnapshot` callback). The key insight: before the first sync event fires, the Realtime snapshot is empty, so `onlineUserIds` is empty, and `presenceAwareUsers` incorrectly forces everyone to offline.

**2. Fix presenceAwareUsers derivation (lines 134-159)**
Replace the current offline-override logic in the `presenceAwareUsers` useMemo:

Current broken logic (lines 145-151):
```typescript
} else {
  if (!user.status || user.status === 'online') {
    derivedStatus = 'offline';
  } else if (user.status === 'away' || user.status === 'busy') {
    derivedStatus = 'offline';
  }
}
```

Fixed logic:
```typescript
} else {
  // Only override to offline if:
  // 1. Presence system is ready (first sync received) AND
  // 2. User's DB status was 'online' (they should be in presence if truly online)
  // Never override away/busy -- those are intentional DB statuses that don't require Realtime presence
  if (isPresenceReady && (!user.status || user.status === 'online')) {
    derivedStatus = 'offline';
  }
  // If isPresenceReady is false, keep the DB status as-is (trust the server)
  // If user.status is 'away' or 'busy', keep it -- these are explicit user choices
}
```

This prevents: (a) mass false-offline during channel-init gap, (b) away/busy users being forcibly downgraded to offline.

**3. Remove the peer leave handler's fire-and-forget POST to /api/users/location**
In the presence `leave` handler (lines 434-443), DELETE the entire `leftUserIds.forEach(...)` block that POSTs `{userId, spaceId: null, offline: true}` to `/api/users/location`. This block causes peer clients to permanently evict users from the DB on any Realtime disconnect (tab switch, network hiccup). The user's own `beforeunload` handler and the server-side cleanup already handle legitimate offline transitions. Other clients should NEVER write location state for a user that is not themselves.

Keep the `queryClient.setQueryData` block above it (lines 426-432) that marks users as offline in the local cache -- that is correct for triggering the UI fade. Only remove the fetch POST loop.

Also export `isPresenceReady` in the return object so downstream consumers can check readiness if needed.

Two targeted changes to AvatarGroup.tsx:

**1. Cancel fade timeout when a user comes back online**
In the useEffect that detects status changes (lines 65-141), AFTER the block that handles users transitioning TO offline (lines 98-123), add a new block that handles users transitioning FROM offline back to non-offline. For each user currently in `exitingUserIds` whose new status is NOT 'offline':
- Clear the existing timeout from `offlineExitTimeoutsRef`
- Remove the user from `exitingUsersRef`
- Remove the user from `exitingUserIds` state

Logic:
```typescript
// Cancel fade for users that came back online
users.forEach((user) => {
  if (exitingUserIds.has(user.id) && user.status !== 'offline') {
    const existingTimeout = offlineExitTimeoutsRef.current.get(user.id);
    if (existingTimeout) {
      clearTimeout(existingTimeout);
      offlineExitTimeoutsRef.current.delete(user.id);
    }
    exitingUsersRef.current.delete(user.id);
    setExitingUserIds((prev) => {
      const next = new Set(prev);
      next.delete(user.id);
      return next;
    });
  }
});
```

Place this block BEFORE the offline-transition block (before line 98) so that a user who went offline and immediately came back online in the same render cycle gets their fade cancelled first, then the offline block won't re-add them because `user.status !== 'offline'` will be true.

**2. Guard against re-adding already-fading users**
In the offline-transition block (line 98-123), the condition `!prevUser || prevUser.status === 'offline' || user.status !== 'offline'` already returns early, but add an additional guard: skip users already in `exitingUserIds` to prevent resetting their fade timer on every render cycle:

```typescript
if (!prevUser || prevUser.status === 'offline' || user.status !== 'offline') {
  return;
}
// Skip if already fading -- do not reset the timer
if (exitingUserIds.has(user.id)) {
  return;
}
```

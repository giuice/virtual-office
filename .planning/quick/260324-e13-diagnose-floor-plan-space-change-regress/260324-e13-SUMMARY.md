## Root Cause Analysis

### The Immediate Bug
`handleEnterSpace` (floor-plan.tsx:248-262) called `saveLastSpace(space.id)` which updates localStorage and triggers the `useLastSpace` effect. But the effect has a guard at useLastSpace.ts:370:

```typescript
if (currentUser.currentSpaceId && currentUser.currentSpaceId !== context.spaceId) {
  return; // Prevents auto-placement from moving users already in a space
}
```

This guard is correct for AUTO-PLACEMENT (reconnect, first-load) but blocks MANUAL user clicks because `currentSpaceId` is still the old space when the effect fires. Result: no API call → DB not updated → realtime overwrites → avatar fades back.

**Root architectural problem: 4 sources of truth for "user's current space"**

1. **UI state** — `selectedSpace`, `highlightedSpaceId` (React state in FloorPlan)
2. **localStorage** — `lastSpaceId` (persisted across reloads)
3. **Query cache** — `currentSpaceId` in TanStack Query (optimistic updates)
4. **Database** — `users.current_space_id` (authoritative, synced via realtime)

Each fix addressed one synchronization gap but left others open. The `useLastSpace` hook accumulated guards designed for auto-placement that conflict with manual user actions.

### Recommendations to Prevent Recurrence

1. **Separate auto-placement from manual moves:** The `useLastSpace` effect should only handle initial/reconnect placement. Manual space changes should go through `updateLocation()` directly.
2. **Single entry point for space changes:** All space changes (manual click, auto-placement, grace rejoin) should funnel through one function that updates ALL 4 sources of truth atomically.
3. **Reduce hook complexity:** The `useLastSpace` hook has grown to ~410 lines with multiple refs, guards, and retry logic. Consider splitting into `useAutoPlacement` (reconnect/first-load) and keeping `saveLastSpace` as a simple localStorage helper.
